## 1. 题目分析

- **复述题意**：给定一个字符串数组，要求将所有互相为“字母异位词”（即字符种类和数量完全相同，仅顺序不同）的字符串分组，返回分组后的字符串数组列表。
- **模型抽象**：这道题的标准分类是**哈希表（Hash Table）**。核心在于如何为一组“字母异位词”设计一个**唯一且相同**的哈希键（Key）。
- **约束分析**：假设标准 LeetCode 约束为 $N \le 10^4$，最大字符串长度 $K \le 100$。所有字符均为小写英文字母。
  - 因为 $N=10^4$，$O(N^2)$ 的暴力两两比较显然会超时（$10^8$ 次字符串比较），我们必须寻求总体 $O(N \log N)$ 或 $O(N \cdot K)$ 的线性或对数级解法。
- **边界与澄清点**：
  - **空数组**：输入 `[]` 应返回 `[]`。
  - **空字符串**：输入 `[""]` 或 `["",""]` 应能正确分组返回 `[[""]]` 或 `[["",""]]`。
  - **大整数乘积溢出**：若采用数值哈希需要考虑长字符串累乘/累积带来的 JS 数值位超限 `Number.MAX_SAFE_INTEGER` 溢出。
- **总体策略**：为了保证正确性和覆盖应用场景，我将提供三种“本质不同”的解法：首先是易于实现、符合人脑直觉的**“排序关联法”**；然后是工业界中最高效的最优解**“字母计数哈希法”**，将单次计算压缩至严格 $O(K)$；最后，作为资深算法选手拔高视野的补充方案，提供基于算术基本定理的**“质数连乘哈希法”**，并结合 `BigInt` 演示如何在 TypeScript 中规避数值越界灾难。

## 解法 1：排序作为特征键（排序法）

### 思路

- **核心逻辑**：字母异位词的本质是字符组合无序相同。若将两个互为字母异位词的字符串内部字符进行字典序重排，它们必然生成形式唯一且绝对相等的字符串。因此，我们遍历数组中的每个项，将其转化为排序后的字符串来作 Map 的 Key，把原始字符串塞入该 Key 对应的 Value（数组）中即可分类。
- **TS 细节**：
  - 在 JavaScript / TypeScript 体系中，字符串是不可变的且没有自带排序原语。必须通过将其打散成数组比如 `[...s]`（处理多字节 emoji 安全）或 `s.split('')`转数组，调用原生 `.sort()` 后再用 `.join('')` 重回字符串。
  - 使用 `Map<string, string[]>` 承担映射核心，由于 TS 的 `Map` 对于 Primitive 类型比如 string 非常友好且避免了对象 key 的原型链污染隐患，`map.get/set` 的性能是 $O(1)$ 的最佳实践。
  - 排序法虽不是时间下限，但在实际 V8 内部，由于约束 $K \le 100$，短字符串切片与合并反而有时能比纯算术跑出更惊艳的耗时（归功于底层内部的高度原生优化）。

### 代码实现 (LeetCode 提交版)

```ts
/**
 * 复杂度分析：
 * 时间复杂度：O(N * K log K) —— N 是字符串的个数，K 是字符串的最大长度。主要耗时在于对每个字符长度至多 K 的字符串进行数组字典排序，单趟 O(K log K)。
 * 空间复杂度：O(N * K) —— O(N) 用来存储总长度等于输入项数的输出数组结构，外加 O(K) 是单次循环产生的切割、排序的临时空间，Map 本身承载全部字符数据的内存映射也是 O(N * K)。
 */

function groupAnagrams(strs: string[]): string[][] {
  if (strs.length === 0) return [];

  const map = new Map<string, string[]>();

  // 大数据量处理，优先使用原生迭代减少闭包/高阶方法调用（如 forEach）的损耗
  for (const s of strs) {
    // 按字典序排序取得统一状态签名字串
    const key = s.split('').sort().join('');

    // 如果存在就 push，不存在则带原值初始化一个新数组
    const group = map.get(key);
    if (group !== undefined) {
      group.push(s);
    } else {
      map.set(key, [s]);
    }
  }

  return Array.from(map.values());
}
```

## 解法 2：字符频次数组作为标识（计数哈希法）

### 思路

- **核心逻辑**：由于题目中**仅包含小写字母**，这是极为强烈的降维暗示。我们并不需要通过真正排序来得到“同构”凭证，我们只需统计各个字符串其 26 位 a-z 小写英文字母出现的次数。一旦两根字符串每个字母出现频次一模一样，必互为异位词。我们将这个长度为 26 的计数阵列拼接成类似 `0,1,0,...2` 样式的定长模板作为 Key 进行 Map 归集，以此把排序操作省略。
- **TS 细节**：
  - 在 Node / 常规引擎中不建议直接把普通 `Array` 或者 `Object` 当 Key（这基于强引用判断且无法做值等）。使用带分隔符比如 `,` 分隔拼接的字符串是最安稳的序列化值。或者 `Int32Array.prototype.join` 是一种十分高效原生序列化。
  - 面向性能进行工程优化时，绝对避免在每一次外部大外层 `for` 循环内进行 `new Array(26)`，这会导致庞大的堆回收（GC）压力。我会在外提层创建一个 `Int32Array(26)`，然后在内循环前 `count.fill(0)` 极致复用这段连续空间以提速！

### 代码实现 (LeetCode 提交版)

```ts
/**
 * 复杂度分析：
 * 时间复杂度：O(N * K) —— N 为字符串列表总长，K 为字符串最大长。遍历只需跑一圈累加次数，不再排序；join(26长度的固定数组) 被视作 O(1)。严格逼近理论时间下限。
 * 空间复杂度：O(N * K) —— 返回数组开销，Map 本体开销不变，外加极少几乎静态 O(1) 的 26位 Int32Array 常数数组消耗。
 */

function groupAnagrams(strs: string[]): string[][] {
  if (strs.length === 0) return [];

  const map = new Map<string, string[]>();

  // GC 避免：在外部声明一个 TypedArray 用于所有字符串共用
  const count = new Int32Array(26);

  for (let i = 0; i < strs.length; i++) {
    const s = strs[i];

    // 将计数器清零
    count.fill(0);

    // 快速 ASCII 值映射，97 对应 'a' 的码位
    for (let j = 0; j < s.length; j++) {
      count[s.charCodeAt(j) - 97]++;
    }

    // 拼装字符序列化的 hash 特征作为键
    const key = count.join(',');

    const group = map.get(key);
    if (group !== undefined) {
      group.push(s);
    } else {
      map.set(key, [s]);
    }
  }

  return Array.from(map.values());
}
```

## 解法 3：质数乘积定理特性（算数基本定理哈希法）

### 思路

- **核心逻辑**：数学领域的**算术基本定理**明确道：任何大于 1 的自然数都能唯一地分解成有限个质数的连乘积。如果我们将 'a','b','c' 分别用固定映射的前 26 位素数 $\{2, 3, 5, 7, 11... 101\}$ 替代。对于任何字母异位词群体如 `abc` 和 `bac` ，由于乘法交换律，它们代表的质数之积均为 $2 \times 3 \times 5 = 30$ 绝对一致！没有相同字符集的字符串绝对配凑不出同一个积。
- **TS 细节**：
  - **严重 BigInt 警示**：若 $100$ 长度全为 `z`（映射最后一位素数 101），其连乘会是一个天文数值 $101^{100}$，这远超 JS `Number.MAX_SAFE_INTEGER` 的安全界限 $2^{53}-1$（约 $9 \times 10^{15}$）。传统乘法必发“大数坍塌精度丢失现象”，把不同的长词错配！
  - 于是高级 TS 工程师立刻请出内置的大整数基础类**`BigInt`**(后缀配 `n` 标识字面量)，不仅绝无溢出，`Map` 更对 `BigInt` 实现完美的支持而无任何转换字符串的开销。此解法非常炫技且能在约束极小且不能分配数组的新苛刻环境中大放异彩。

### 代码实现 (LeetCode 提交版)

```ts
/**
 * 复杂度分析：
 * 时间复杂度：O(N * K) —— 从循环层面上和解法 2 旗鼓相当。但 BigInt 的底层大数算法开销导致它的真实运行常数（C常量）比基础 Array 处理要大，所以仅看 O 标记虽完美但有工程耗损。
 * 空间复杂度：O(N * K) —— Map 和返回数组依然占大头空间。单次的 Key 不是 String 而是变长的 BigInt。没有创建任何动态数组。
 */

function groupAnagrams(strs: string[]): string[][] {
  if (strs.length === 0) return [];

  // 前 26 个质数的 BigInt 静态表映射
  const primes = [
    2n,
    3n,
    5n,
    7n,
    11n,
    13n,
    17n,
    19n,
    23n,
    29n,
    31n,
    37n,
    41n,
    43n,
    47n,
    53n,
    59n,
    61n,
    67n,
    71n,
    73n,
    79n,
    83n,
    89n,
    97n,
    101n,
  ];

  const map = new Map<bigint, string[]>();

  for (let i = 0; i < strs.length; i++) {
    const s = strs[i];
    let key = 1n; // 累积素数乘积（基始化为大数 1n）

    for (let j = 0; j < s.length; j++) {
      key *= primes[s.charCodeAt(j) - 97];
    }

    const group = map.get(key);
    if (group !== undefined) {
      group.push(s);
    } else {
      map.set(key, [s]);
    }
  }

  return Array.from(map.values());
}
```

## 解法对比与总结

1. **适用场景**：
   - **解法 1 (排序特征法)**：逻辑异常清晰。若面试无进一步深度要求 或 面临包含大一统 Unicode 生僻字和全小写之外的多样集，则排序法无需预估基数桶长度，普适性甚至更强！
   - **解法 2 (计数哈希法)**：现代 Web 高频场景对性能最妥贴的做法！对已知字符域宽度的题目效果卓越，由于避免了大量底层库内置的序列交换排序压力，$O(N \cdot K)$ 令其具有抗击大规模数据输入的绝佳压舱石。
   - **解法 3 (质数哈希法)**：在没有额外 `Array` 和由于严苛规定甚至连 String 也不能 `join` 生成的“神仙打架局”，利用数学魔法做破局。但因为依赖 `BigInt` 引擎性能，更注重算法逻辑传达而非极致跑分。

2. **工程权衡**：
   - **迭代及避免高阶函数**：大规模数据前沿尽量避免 `.map()`, `.forEach()`，原生的 `for(let i=0..)` 取用 `s.charCodeAt` 可逃避巨大的匿名闭包函数压栈出栈时耗。
   - **对象可复用性与 GC**：解法 2 在外层实例化 `count = new Int32Array` 通过重复抹零 `count.fill(0)` 规避 V8 处理万级短生命周期新生代对象的 GC 长停顿。这是一个极其成熟有魅力的资深工程素养体现。

3. **关键测试样例**：
   - **用例 1 (通用标准覆盖)**
     - 输入：`strs = ["eat", "tea", "tan", "ate", "nat", "bat"]`
     - 期望输出：`[["eat","tea","ate"], ["tan","nat"], ["bat"]]`
     - 测试目标：覆盖普遍打乱状态及验证 Map 分组收集和新项隔离表现完好。
   - **用例 2 (最简空集约束及极值验证)**
     - 输入：`strs = []`
     - 期望输出：`[]`
     - 测试目标：验证无循环触及返回边界。
   - **用例 3 (空字符串复测组化)**
     - 输入：`strs = ["", "b", ""]`
     - 期望输出：`[["", ""], ["b"]]`
     - 测试目标：空字符串能被视作合法实体并组内相互聚合。对解法二的频次全为0和解法三质数乘积始终为 1n 提供鲁棒印证。
   - **用例 4 (全同字符及溢出攻击测试)**
     - 输入：`strs = ["z"..重叠 100遍.., "z"..重叠 100遍..]`
     - 期望输出：`[["zz...z", "zz...z"]]`
     - 测试目标：考验大项串长度时，算法执行是否突破 JS 极限或被判定不同组，解法三中不用 `BigInt` 则由于安全实数尽头发生灾难级碰撞或浮点近似进而失效。
   - **用例 5 (相似 ASCII 总数诱导混淆测试)**
     - 输入：`strs = ["ac", "bb"]`
     - 期望输出：`[["ac"], ["bb"]]`
     - 测试目标：有些初级菜鸟只对内部 ASCII 原生加和作 Hash！他们得到的特征皆为 $a+c=1+3=4, b+b=2+2=4$，从而混在一起返回报错！确保三版解法全数抵御此类投机简化。
