## 1. 题目分析

- **复述题意**：给定一个整数数组 `nums` 和一个目标值 `target`，找到数组中两个不同元素的下标，使得它们的和等于 `target`。需返回这两个下标。
- **模型抽象**：这是一个经典的「查找问题」。本质是在数组中寻找 `val` 和 `target - val` 是否同时存在。模型可选：暴力枚举、哈希表查找（空间换时间）、排序后双指针（需处理下标）。
- **约束分析**：
  - $N \le 10^4$：
    - $O(N^2)$ 解法约为 $10^8$ 次操作。在 JavaScript/V8 中，简单加法运算 $10^8$ 次可能耗时约 100-300ms，通常能通过 LeetCode，但属于“压线过”，不是最优。
    - $O(N)$ 或 $O(N \log N)$ 是更安全且符合面试预期的复杂度。
  - 数值范围 $-10^9 \le nums[i] \le 10^9$：
    - 加和最大可能为 $2 \times 10^9$，远小于 JS `Number.MAX_SAFE_INTEGER` ($9 \times 10^{15}$)。因此**不需要 BigInt**，常规 `number` 即可。
- **边界与澄清点**：
  - **唯一解**：题目保证“只会存在一个有效答案”，无需处理多解或无解情况。
  - **自身重复**：题目要求“不能使用两次相同的元素”，即下标 $i \neq j$。
  - **负数**：输入包含负数，无法通过“大于 target 就剪枝”来优化（除非数组全正）。
- **总体策略**：
  1.  **解法 1（暴力法）**：作为兜底方案，双重循环，演示最基础逻辑。
  2.  **解法 2（哈希表）**：**最优解**。利用 Map 记录 `(值 -> 下标)`，实现 $O(N)$ 时间复杂度。
  3.  **解法 3（排序+双指针）**：虽然时间复杂度 $O(N \log N)$ 劣于哈希法，但它是“双指针”技巧的典型应用（常用于 3Sum、4Sum 或输入已排序场景），且在面试中常作为“如果空间要求更严苛或不能用哈希”的备选讨论。

## 解法 1：暴力枚举法 (Brute Force)

### 思路

- **核心逻辑**：使用两层循环。外层循环遍历 `i`，内层循环遍历 `j`（从 `i + 1` 开始），检查 `nums[i] + nums[j] === target`。
- **TS 细节**：
  - 使用基础 `for` 循环而非 `forEach/map`，避免回调函数创建和上下文绑定的微小开销（在 $O(N^2)$ 下这点开销可忽略，但为了习惯）。
  - 直接返回字面量数组 `[i, j]`。

### 代码实现 (LeetCode 提交版)

```ts
/**
 * 复杂度分析：
 * 时间复杂度：O(N^2) —— 包含两层嵌套循环，最坏情况下执行 N*(N-1)/2 次比较。
 * 空间复杂度：O(1) —— 仅使用常数级额外空间存储循环变量。
 */

function twoSum_BruteForce(nums: number[], target: number): number[] {
  const n = nums.length;
  // 第一层循环：选择第一个数
  for (let i = 0; i < n; i++) {
    // 第二层循环：选择第二个数（从 i+1 开始，避免重复和自身）
    for (let j = i + 1; j < n; j++) {
      if (nums[i] + nums[j] === target) {
        return [i, j];
      }
    }
  }
  // 题目保证有解，理论上不会走到这里，但 TS 规范需要返回空数组或抛错
  return [];
}
```

## 解法 2：哈希表法 (Hash Map) —— **推荐/最优解**

### 思路

- **核心逻辑**：
  - 利用“空间换时间”。我们在遍历数组时，对于当前元素 `x`，我们需要寻找是否存在一个 `target - x` 已经在之前出现过。
  - 维护一个哈希表 `Map<数值, 下标>`。
  - **一次遍历 (One-pass)**：遍历数组，先查表，若 `target - nums[i]` 存在，则直接返回；否则将 `nums[i]` 及其下标存入表中。
- **TS 细节**：
  - **Map vs Object**：推荐使用 `Map`。在 V8 引擎中，当 Key 为频繁增删的整数（且非连续的小整数）时，`Map` 通常比 `Object` 性能更稳健，且避免了 `Object` 原型链上的杂质。
  - **类型安全**：明确定义 `Map<number, number>`。

### 代码实现 (LeetCode 提交版)

```ts
/**
 * 复杂度分析：
 * 时间复杂度：O(N) —— 只需遍历一次数组，Map 的查找与插入操作平均为 O(1)。
 * 空间复杂度：O(N) —— 最坏情况下需要存储 N-1 个元素到哈希表中。
 */

function twoSum(nums: number[], target: number): number[] {
  // 记录已经遍历过的数值及其对应的下标
  // Key: nums[i] (数值), Value: i (下标)
  const prevNumMap = new Map<number, number>();

  for (let i = 0; i < nums.length; i++) {
    const currentNum = nums[i];
    const complement = target - currentNum;

    // 检查补数是否在之前已经出现过
    if (prevNumMap.has(complement)) {
      // 如果存在，直接返回 [补数的下标, 当前下标]
      // 此时 map.get(complement) 一定不为 undefined，使用断言或直接获取
      return [prevNumMap.get(complement)!, i];
    }

    // 将当前数值存入 Map，供后续查找
    prevNumMap.set(currentNum, i);
  }

  return [];
}
```

## 解法 3：排序 + 双指针 (Sorting + Two Pointers)

### 思路

- **核心逻辑**：
  - 双指针法通常用于**有序数组**（如 LeetCode 167 题）。
  - 本题数组无序，如果直接排序会打乱原始下标。因此，必须先创建一个包含 `{ value, index }` 的结构，对该结构按 `value` 排序。
  - 排序后，定义 `left = 0`, `right = N - 1`。
    - 若 `nums[left] + nums[right] > target`，说明和太大，`right--`。
    - 若 `nums[left] + nums[right] < target`，说明和太小，`left++`。
    - 相等则找到答案，返回对应的原始下标。
- **TS 细节**：
  - **结构化数据**：为了保持原始下标，可以生成一个 `[number, number][]` 的二维数组 `pairs`，其中 `pairs[i] = [数值, 原始下标]`。
  - **性能优化**：虽然这题主要为了演示算法思想，但在工程中，创建大量小对象或数组会增加 GC 压力。如果极致优化，可以用两个 `Int32Array` 分别存值和下标，手动同步排序（但在 JS 中手写快排较繁琐，这里使用原生 `sort` 配合元组）。

### 代码实现 (LeetCode 提交版)

```ts
/**
 * 复杂度分析：
 * 时间复杂度：O(N log N) —— 主要消耗在数组排序上。双指针遍历仅需 O(N)。
 * 空间复杂度：O(N) —— 需要创建一个额外的数组来存储数值与原始下标的映射关系。
 */

function twoSum_TwoPointers(nums: number[], target: number): number[] {
  const n = nums.length;

  // 1. 构建带有原始下标的数组：[[val, index], [val, index], ...]
  // 这里使用元组 (tuple) 以符合 TS 规范
  const pairs: [number, number][] = new Array(n);
  for (let i = 0; i < n; i++) {
    pairs[i] = [nums[i], i];
  }

  // 2. 按照数值大小进行升序排序 O(N log N)
  pairs.sort((a, b) => a[0] - b[0]);

  // 3. 双指针寻找目标和 O(N)
  let left = 0;
  let right = n - 1;

  while (left < right) {
    const sum = pairs[left][0] + pairs[right][0];

    if (sum === target) {
      // 找到答案，返回原始下标
      return [pairs[left][1], pairs[right][1]];
    } else if (sum < target) {
      // 和偏小，左指针右移，让和变大
      left++;
    } else {
      // 和偏大，右指针左移，让和变小
      right--;
    }
  }

  return [];
}
```

## 解法对比与总结

### 1. 适用场景

- **解法 2 (哈希表)**：**首选通用方案**。适用于 $N$ 较大且无序的场景，时间复杂度最低 $O(N)$。
- **解法 1 (暴力)**：仅适用于 $N$ 非常小（如 $N < 100$）或不允许使用额外空间的场景（虽然严格来说 $O(1)$ 空间也很难，因为递归栈或输入修改也是成本）。
- **解法 3 (排序双指针)**：适用于**数组已经被排序**的情况（此时无需额外空间，直接 $O(N)$ 时间，$O(1)$ 空间）。或者作为“三数之和”、“四数之和”的前置思维（因为哈希表处理 3Sum/4Sum 去重很麻烦，排序+双指针更优）。

### 2. 工程权衡

- **Map vs Object**：本题中 Key 是 Int，`Map` 在 V8 中会被优化为 Hash Table（甚至某些情况下是线性探测），性能优异。

  > 为什么Map 在 V8中为什么性能优异？
  >
  > `Map` 在 V8 里性能优异，主要因为它是“为动态哈希映射设计的专用数据结构”，对大量不稳定键的插入/查询/删除更稳定；而对象的性能优势来自 Hidden Class 的属性访问快路径，把对象当哈希表会更容易退化到 dictionary mode 并破坏 JIT 优化。

- **GC 压力**：解法 3 创建了 `N` 个小数组 `[val, idx]`，这在 Node.js 高并发场景下会对 GC 造成瞬间压力。解法 2 仅创建一个 Map，相对较优。
