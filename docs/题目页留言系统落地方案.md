# 题目页留言系统落地方案

## 1. 方案结论

本项目题目页留言功能的最终选型为：

- 评论存储：`GitHub Discussions`
- 评论承载组件：`giscus`
- 评论线程维度：每道题一条独立讨论线程
- 接入方式：题目页路由层挂载，客户端懒加载

本文档只描述这一条正式方案，不再展开其他存储路线。

---

## 2. 目标

目标如下：

- 在每个算法题页面增加留言区
- 用户可以查看其他人的留言
- 用户通过 GitHub 账号登录后可参与留言
- 不自建数据库
- 不破坏当前题目页静态化与首包优化成果

非目标如下：

- 不支持游客匿名留言
- 不自建站内账号体系
- 不自建评论审核后台
- 不在本阶段实现“点赞排行、精选评论、站内通知”

---

## 3. 为什么选 GitHub Discussions

这个项目当前最适合把评论存到 `GitHub Discussions`，原因很直接：

- 当前仓库没有数据库、登录态、用户系统、审核后台
- 题目页已经静态化，评论不应该迫使页面改成动态 SSR
- 项目刚处理过题目页 3MB 首包问题，评论系统必须尽量轻量
- 算法站点的目标用户普遍可以接受 GitHub 登录

`GitHub Discussions + giscus` 的收益：

- 不需要自建数据库
- 不需要自建评论接口
- 不需要自建登录系统
- 评论天然持久化并可跨用户共享
- GitHub 自带基础账号、风控、通知和管理能力

---

## 4. 当前项目的约束

本方案必须服从当前仓库的几个关键约束。

### 4.1 题目页当前是静态生成

文件：

- `src/app/problems/[category]/[problem]/page.tsx`

当前已明确配置：

```ts
export const dynamic = 'force-static';
export const dynamicParams = false;
export const revalidate = false;
```

这意味着：

- 题目页不能因为评论而改成依赖服务端实时渲染
- 评论区应当作为客户端附加模块存在

### 4.2 可视化本体已经做了客户端懒加载

文件：

- `src/components/visualizer/ProblemVisualizerClient.tsx`

当前 `GenericVisualizer` 是动态导入的。这条边界不能被评论逻辑污染，否则容易把题目页首包重新做大。

### 4.3 可视化布局是固定视口高

文件：

- `src/components/shared/VisualizerLayout.tsx`

当前布局使用固定高度和 `overflow-hidden`，说明它适合可视化主界面，不适合直接在内部拼接长列表讨论区。

结论：

- 评论区不要塞进 `GenericVisualizer`
- 评论区也不要塞进 `VisualizerLayout`
- 评论区应该挂在题目页路由层，在可视化模块之后

### 4.4 当前 CSP 默认不允许 giscus

文件：

- `next.config.ts`

当前 CSP 没有为 `giscus.app` 放行脚本和 iframe，因此接入前必须把这个约束写进实施清单。

---

## 5. 总体架构

推荐架构如下：

1. 题目页继续保持静态生成
2. 可视化播放器继续按现有方式懒加载
3. 评论区作为题目页下方的独立模块
4. 评论区只在客户端挂载
5. 评论线程稳定映射到 `GitHub Discussions`

推荐页面结构：

```tsx
<div className="bg-slate-950">
  <TypeScriptCodeHint />
  <ProblemVisualizerClient category={category} problemSlug={problem} />
  <ProblemCommentsSection category={category} problem={problem} />
</div>
```

这个结构的意义：

- 评论逻辑和可视化逻辑完全解耦
- 评论区处于首屏以下，更容易做懒加载
- 不破坏现有移动端和桌面端可视化布局

---

## 6. 评论线程映射规则

评论必须按题目维度稳定映射，不建议直接使用完整 URL。

推荐规则：

```txt
problem:${category}/${problem}
```

示例：

```txt
problem:dynamic-programming/word-break
problem:trie/word-search-ii
problem:array/two-sum
```

这样做的好处：

- 域名变化不会导致评论线程丢失
- 预览环境、本地环境、生产环境都能映射到同一题
- 未来题解页也可以共用同一讨论线程

## 7. GitHub Discussions 侧准备

在接入代码前，先完成 GitHub 侧准备。

### 7.1 开启 Discussions

要求：

- 目标仓库已开启 `GitHub Discussions`
- 仓库对目标用户可访问

### 7.2 创建评论分类

建议在仓库 Discussions 中创建一个专门分类，例如：

- `Problem Comments`

原因：

- 与一般社区讨论分离
- 便于后期管理和检索

### 7.3 获取 giscus 配置参数

需要从 `giscus` 配置页拿到以下参数：

- `repo`
- `repoId`
- `category`
- `categoryId`

推荐使用：

- `mapping = specific`
- `term = problem:${category}/${problem}`

参考：

- https://giscus.app/zh-CN

## 8. 建议新增的文档内实施对象

本阶段只写方案，不修改业务代码。后续实施时建议新增以下对象：

```txt
src/
  components/
    comments/
      ProblemCommentsSection.tsx
      GiscusComments.tsx
  lib/
    comments/
      problem-comment-thread.ts
```

### 8.1 `problem-comment-thread.ts`

职责：

- 统一生成题目评论线程 key

建议接口：

```ts
export function getProblemCommentTerm(category: string, problem: string): string {
  return `problem:${category}/${problem}`;
}
```

### 8.2 `ProblemCommentsSection.tsx`

职责：

- 评论区外壳
- 展示标题与说明
- 做可见性检测
- 到达视口后再挂载真正评论组件

### 8.3 `GiscusComments.tsx`

职责：

- 客户端注入 `giscus` 脚本
- 负责将当前题目映射到对应 Discussion 线程

## 9. 环境变量设计

后续实施时建议使用以下环境变量：

```env
NEXT_PUBLIC_GISCUS_REPO=owner/repo
NEXT_PUBLIC_GISCUS_REPO_ID=xxxxxxxx
NEXT_PUBLIC_GISCUS_CATEGORY=Problem Comments
NEXT_PUBLIC_GISCUS_CATEGORY_ID=yyyyyyyy
NEXT_PUBLIC_GISCUS_LANG=zh-CN
```

说明：

- 都使用 `NEXT_PUBLIC_` 前缀，因为 `giscus` 运行在客户端
- 这些变量建议补充到 `.env.example`

## 10. 页面接入策略

### 10.1 题目页

接入文件：

- `src/app/problems/[category]/[problem]/page.tsx`

接入位置：

- `ProblemVisualizerClient` 之后

理由：

- 不干扰题目页校验逻辑
- 不污染可视化 chunk
- 评论区自然在首屏以下

### 10.2 题解页

接入文件：

- `src/app/problems/[category]/[problem]/solution/page.tsx`

推荐策略：

- 题解页与题目页共用同一线程

原因：

- 用户讨论的是同一道题，不需要拆成两条讨论流
- 有利于内容沉淀

如果第一阶段只想在题目页展示评论，也建议线程 key 仍然按题目统一设计，避免后续重复迁移。

---

## 11. giscus 接入参数建议

推荐参数如下：

```txt
data-mapping = specific
data-strict = 1
data-reactions-enabled = 1
data-emit-metadata = 0
data-input-position = bottom
data-theme = dark
data-lang = zh-CN
```

说明如下：

- `mapping = specific`
  - 明确按题目 key 绑定线程
- `strict = 1`
  - 精确匹配 `term`
- `reactions-enabled = 1`
  - 可以保留 GitHub Discussions 的反应能力
- `theme = dark`
  - 当前项目整体偏深色，第一阶段直接固定深色最稳

---

## 12. 懒加载要求

评论模块必须懒加载，这是本方案的硬约束。

推荐做法：

- 题目页先渲染评论区骨架
- 使用 `IntersectionObserver`
- 当评论区接近视口时再真正挂载 `giscus`

原因：

- 避免首屏立刻加载第三方评论资源
- 避免影响题目页首包与首屏交互
- 与当前项目的性能优化方向一致

推荐触发策略：

- `rootMargin: '320px 0px'`

这样用户接近评论区时就开始加载，体验更平滑。

---

## 13. CSP 调整要求

当前项目的 `Content-Security-Policy` 需要为 `giscus` 放行。

涉及文件：

- `next.config.ts`

至少需要覆盖以下能力：

- `script-src` 允许 `https://giscus.app`
- `frame-src` 允许 `https://giscus.app`

建议目标形态：

```ts
{
  key: 'Content-Security-Policy',
  value: [
    "default-src 'self'",
    `script-src ${scriptSrc} https://giscus.app`,
    "style-src 'self' 'unsafe-inline'",
    "img-src 'self' data: blob:",
    "font-src 'self' data:",
    "connect-src 'self' https:",
    "frame-src https://giscus.app",
    "frame-ancestors 'none'",
  ].join('; '),
}
```

注意：

- 不要把 CSP 粗暴放宽成 `*`
- 评论能力只需要为 `giscus` 开必要白名单

## 14. UI 方案

第一阶段推荐做成：

- 题目页下方独立讨论卡片

不推荐第一阶段做成：

- 右侧抽屉
- 全局浮层
- 嵌入到可视化主布局内部

原因：

- 当前布局本来就是为全屏可视化设计
- 题目页下方卡片方案最稳，改动最少
- 后续如果需要再把评论区升级为抽屉，不影响线程映射规则

视觉建议：

- 延续现有 `slate` 深色体系
- 标题统一使用“题目讨论”
- 副文案说明“使用 GitHub 登录后留言”

## 17. 风险与注意事项

### 17.1 用户必须有 GitHub 账号

这是方案天然限制，不是 bug。

### 17.2 评论管理主要依赖 GitHub 平台能力

如果未来需要更强的站内治理能力，需要再升级方案。

### 17.3 不能把评论写回当前仓库静态内容

否则会和当前 SSG 架构冲突，并带来部署负担。

### 17.4 评论模块不能破坏现有性能边界

这一点必须在实施和回归测试中重点盯住。

## 18. 上线检查清单

上线前逐项确认：

1. 仓库已开启 `GitHub Discussions`
2. 已创建 `Problem Comments` 分类
3. 已拿到 `repoId` 和 `categoryId`
4. 生产环境变量已配置
5. `CSP` 已放行 `giscus.app`
6. 评论区已做懒加载
7. 题目页构建结果未出现首包回升
8. 题目页评论线程映射规则已固定为 `problem:${category}/${problem}`
9. 题解页是否共用线程已确认

## 19. 最终建议

本项目的留言系统应当按以下方式落地：

- 存储统一放在 `GitHub Discussions`
- 前端展示统一使用 `giscus`
- 评论线程统一按题目 slug 映射
- 评论区挂在题目页路由层
- 评论组件必须懒加载
- 不修改题目页静态化策略

这条路线在当前项目里成本最低、风险最小、维护最轻，同时能满足“每个人都能留言并查看别人的留言”的核心目标。
