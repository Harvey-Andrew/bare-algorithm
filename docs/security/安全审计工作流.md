## 2. 审计范围

当前安全审计由两部分组成：

1. 自动化审计（脚本 `scripts/security-audit.mjs`）
2. 人工审计（接口滥用面、输入校验、XSS 面、配置回归）

自动化脚本用于基线检查，不替代人工业务安全评估。

## 3. 自动化审计流程

### 3.1 四项固定检查

1. 依赖漏洞扫描  
   命令：`pnpm audit --registry https://registry.npmjs.org --json`

2. 许可证合规检查  
   命令：`npx -y license-checker --production --json`

3. 安全响应头检查  
   静态检查 `next.config.ts` 是否包含：

- `X-Frame-Options`
- `X-Content-Type-Options`
- `Referrer-Policy`
- `Permissions-Policy`
- `Strict-Transport-Security`
- `Content-Security-Policy`

4. 敏感文件保护检查  
   检查 `.gitignore` 是否包含：

- `.env*`
- `*.pem`
- `*.key`
- `.vercel`

并额外扫描根目录 `.env*`（排除 `.env.example`）写入报告。

### 3.2 评分模型

```text
score = 100
      - critical * 15
      - high * 5
      - moderate * 2
      - low * 1
      - blocked_license * 10
      - missing_header * 10
      - unprotected_sensitive_pattern * 10
```

### 3.3 输出产物

每次执行产出：

- `docs/security/reports/<timestamp>/summary.html`
- `docs/security/reports/<timestamp>/summary.md`
- `docs/security/reports/<timestamp>/summary-data.json`

## 4. 人工审计清单（执行模板）

### 4.1 接口滥用与防刷

- 是否存在匿名写接口
- 是否具备速率限制
- 是否具备人机挑战校验
- 是否具备来源约束（可选辅助）

### 4.2 输入与边界

- 是否有运行时校验（不能只依赖 TS 类型）
- 字段长度、枚举、结构是否受控
- 是否有服务端 secret 泄露风险

### 4.3 前端注入风险

- 是否使用 `dangerouslySetInnerHTML`
- 输入是否做 escape/sanitize
- 外链打开是否使用 `noopener/noreferrer`

### 4.4 配置与部署

- CSP 是否过宽（`unsafe-inline`/`unsafe-eval`）
- 缓存与响应头是否符合预期
- 安全策略是否在多实例部署下仍然有效

## 5. 常用命令

```bash
# 本地审计
pnpm security:audit

# CI 门禁模式（high/critical 漏洞返回非 0）
pnpm security:audit:ci

# 关键变更后的建议补充检查
pnpm exec tsc --noEmit
pnpm exec eslint src/app/api/issues/route.ts src/components/issues/HomeIssueComposer.tsx
```

## 6. CI 建议

建议 PR 流程至少包含：

1. `pnpm lint`
2. `pnpm test`
3. `pnpm security:audit:ci`

当前脚本不会因为“许可证/安全头/敏感文件模式”直接阻断 CI，这些问题会在报告中暴露，需要人工决策。
