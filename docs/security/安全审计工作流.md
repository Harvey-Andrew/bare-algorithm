# 安全审计工作流说明

本文档描述项目当前可执行的安全审计流程，目标是把「依赖漏洞扫描 → 许可证合规 → 安全配置校验 → 敏感信息保护」固化成标准化操作，并自动生成可视化报告。

> **项目概况**：Algorithm 朴素算法 — Next.js 16 + React 19 + TypeScript，纯静态 SSG 部署于 Vercel。

## 1. 目标

- 自动检测已知 CVE 依赖漏洞，按严重程度分级告警
- 确保所有生产依赖许可证与项目 MIT 兼容，阻断 GPL/AGPL 等传染性许可
- 校验 `next.config.ts` 中 6 项安全响应头不被意外移除
- 验证 `.gitignore` 对敏感文件模式的保护覆盖
- 输出带评分的可视化报告（HTML + Markdown + JSON），留存审计证据

## 3. 四维审计详解

### 3.1 依赖漏洞审计

| 项目     | 说明                                                       |
| -------- | ---------------------------------------------------------- |
| 工具     | `pnpm audit --json`                                        |
| 数据源   | npm Advisory 数据库（`registry.npmjs.org`）                |
| 超时     | 30 秒                                                      |
| 降级策略 | 网络不通时自动使用 `docs/security/audit-raw.json` 缓存数据 |

**漏洞分级与 CI 行为：**

| 等级     | 图标 |  评分扣分 | CI 行为（`--ci` 模式） | 处理要求                 |
| -------- | ---- | --------: | ---------------------- | ------------------------ |
| Critical | 🔴   | -15 分/个 | ❌ 阻断构建            | 立即修复                 |
| High     | 🟠   |  -5 分/个 | ❌ 阻断构建            | 24 小时内修复            |
| Moderate | 🟡   |  -2 分/个 | ⚠️ 仅警告              | 跟踪 Issue，下个迭代修复 |
| Low      | 🔵   |  -1 分/个 | ✅ 通过                | 知悉即可                 |
| Info     | ⚪   |         0 | ✅ 通过                | 可忽略                   |

**报告输出信息：**

- 扫描依赖总数
- 各级别漏洞计数与占比
- 受影响包列表（包名、版本、漏洞数、最高等级、修复版本）
- 漏洞详情表（Advisory ID、CVE、CVSS 分数、标题、链接）

### 3.2 许可证合规检查

| 项目 | 说明                                         |
| ---- | -------------------------------------------- |
| 工具 | `npx -y license-checker --production --json` |
| 超时 | 60 秒                                        |
| 范围 | 仅生产依赖（`--production`）                 |

**许可证分类与判定：**

| 分类             | 许可证                    |  评分扣分 | 判定结果     |
| ---------------- | ------------------------- | --------: | ------------ |
| ✅ 宽松许可证    | MIT, ISC, BSD, Apache-2.0 |         0 | 兼容         |
| ⚠️ 弱 Copyleft   | LGPL-\*                   | 0（警告） | 需关注       |
| ❌ 强 Copyleft   | GPL-2.0, GPL-3.0          | -10 分/个 | 不兼容，阻断 |
| ❌ 网络 Copyleft | AGPL-3.0, SSPL-1.0        | -10 分/个 | 不兼容，阻断 |

**被阻断的许可证完整列表：**

```
GPL-2.0, GPL-2.0-only, GPL-3.0, GPL-3.0-only,
AGPL-3.0, AGPL-3.0-only, SSPL-1.0
```

**报告输出信息：**

- 扫描包总数与合规率
- 许可证分布 Top 10 饼图（HTML 报告）
- 不兼容/需关注的包列表

### 3.3 安全响应头校验

通过静态分析 `next.config.ts` 文件内容，检查以下 6 个必需安全头是否存在：

| 安全头                      | 防护作用           | 评分扣分（缺失时） |
| --------------------------- | ------------------ | -----------------: |
| `X-Frame-Options`           | 防御点击劫持       |             -10 分 |
| `X-Content-Type-Options`    | 阻止 MIME 嗅探     |             -10 分 |
| `Referrer-Policy`           | 控制 Referer 泄露  |             -10 分 |
| `Permissions-Policy`        | 禁用敏感浏览器 API |             -10 分 |
| `Strict-Transport-Security` | 强制 HTTPS         |             -10 分 |
| `Content-Security-Policy`   | 限制资源加载来源   |             -10 分 |

### 3.4 敏感文件保护

检查 `.gitignore` 是否包含以下敏感文件模式：

| 模式      | 说明            | 评分扣分（未保护时） |
| --------- | --------------- | -------------------: |
| `.env*`   | 环境变量文件    |               -10 分 |
| `*.pem`   | SSL/TLS 证书    |               -10 分 |
| `*.key`   | 私钥文件        |               -10 分 |
| `.vercel` | Vercel 部署配置 |               -10 分 |

额外检查：扫描项目根目录是否存在 `.env` 文件（排除 `.env.example`），存在则发出警告。

## 4. 评分系统

### 4.1 计算规则

基础分 100 分，各维度扣分叠加，最终分数限制在 0–100 之间：

```
score = 100
       - critical漏洞数 × 15
       - high漏洞数 × 5
       - moderate漏洞数 × 2
       - low漏洞数 × 1
       - 不兼容许可证数 × 10
       - 缺失安全头数 × 10
       - 未保护敏感文件数 × 10
score = clamp(score, 0, 100)
```

### 4.2 等级判定

| 分数区间 | 等级 | 评价 | 颜色编码  |
| -------- | ---- | ---- | --------- |
| 90–100   | A    | 优秀 | `#0f766e` |
| 80–89    | B    | 良好 | `#16a34a` |
| 60–79    | C    | 一般 | `#d97706` |
| 40–59    | D    | 较差 | `#ea580c` |
| 0–39     | F    | 危险 | `#b91c1c` |

## 5. 命令说明

### 5.1 命令清单

| 命令                     | 说明                                | CI 阻断 |
| ------------------------ | ----------------------------------- | :-----: |
| `pnpm security:audit`    | 执行安全审计并生成报告              |   ❌    |
| `pnpm security:audit:ci` | CI 模式（high+ 漏洞返回非零退出码） |   ✅    |

### 5.2 底层实现

```bash
# security:audit
node scripts/security-audit.mjs

# security:audit:ci
node scripts/security-audit.mjs --ci
```

### 5.3 `--ci` 模式说明

在 `--ci` 模式下，脚本执行完所有 4 项审计后，如果发现 **critical 或 high** 级别漏洞，会以 `process.exit(1)` 退出，导致 CI 流水线失败。其他审计维度的问题（许可证、安全头、敏感文件）均反映在评分中但不阻断 CI。

## 7. 报告产物与解读

### 7.1 查看方式

| 报告类型      | 路径                                                  | 用途                    |
| ------------- | ----------------------------------------------------- | ----------------------- |
| 可视化 HTML   | `docs/security/reports/<timestamp>/summary.html`      | 给评审/团队查看的主报告 |
| Markdown 摘要 | `docs/security/reports/<timestamp>/summary.md`        | PR 描述 / README 引用   |
| 结构化 JSON   | `docs/security/reports/<timestamp>/summary-data.json` | 自动化工具消费          |

### 7.2 HTML 报告关键区域

| 区域             | 内容                                                              |
| ---------------- | ----------------------------------------------------------------- |
| 安全评分卡       | SVG 环形图显示综合评分（0-100），带等级标签（A-F）                |
| 依赖漏洞卡       | 各级别漏洞数量，健康度进度条                                      |
| 许可证合规卡     | 合规包数/总包数，合规率进度条                                     |
| 安全响应头卡     | 已配置/总数，覆盖率进度条                                         |
| 受影响包卡片组   | 每个有漏洞的包独立卡片（版本、漏洞数、最高等级、修复版本）        |
| 漏洞详情表       | Advisory ID（可点击跳转）、包名版本、等级徽标、CVE、CVSS 分、标题 |
| 许可证分布饼图   | Top 10 许可证类型的 SVG 环形分布图 + 图例                         |
| 许可证问题表     | 不兼容/需关注的包名、许可证、类型标签                             |
| 安全头配置网格   | 6 项安全头的已配置/缺失状态                                       |
| 敏感文件保护网格 | 4 项敏感模式的已排除/未排除状态                                   |

### 7.3 Markdown 报告内容

报告包含以下章节：

1. **Overview** — 四维状态汇总表
2. **Vulnerability Counts** — 各级别漏洞数量与占比
3. **Affected Packages** — 受影响包表格
4. **Vulnerability Details** — 逐条漏洞详情
5. **License Summary** — 许可证分布 Top 10
6. **Security Headers** — 安全头配置状态与防护说明
7. **Sensitive File Protection** — 敏感文件保护状态

## 8. 操作指南（SOP）

### 场景 A：本地安全审计（日常）

```bash
pnpm security:audit
```

执行后会依次输出 4 个审计步骤的控制台日志：

```
🔍 [1/4] 执行依赖漏洞审计...
  ✅ 扫描 XXX 个依赖，发现 X 个漏洞
📜 [2/4] 执行许可证合规检查...
  ✅ 扫描 XXX 个包，X 个需关注
🔒 [3/4] 检查安全响应头...
  ✅ 6/6 安全头已配置
🗝️ [4/4] 检测敏感文件...
  ✅ 4/4 敏感文件模式已排除

✨ 审计报告已生成：
   📁 docs/security/reports/<timestamp>
   📄 summary.html  (可视化页面)
   📄 summary.md    (Markdown 摘要)
   📄 summary-data.json (原始数据)
   🏆 安全评分：XX/100（X 评价）
```

然后打开 `docs/security/reports/<timestamp>/summary.html` 查看可视化报告。

### 场景 B：CI 安全门禁

```bash
pnpm security:audit:ci
```

与场景 A 相同流程，但在发现 critical/high 漏洞时以非零退出码终止 CI。

### 场景 C：提交前快速检查

```bash
# 仅检查依赖漏洞（不生成完整报告）
pnpm audit --prod --audit-level=high
```

### 场景 D：需要汇报安全状态

```bash
pnpm security:audit
# 将最新 docs/security/reports/<timestamp>/summary.html 发送评审
```

## 9. CI 集成

### 9.1 现有 CI 中的安全步骤

安全审计通过 `pnpm security:audit:ci` 命令集成到 CI 流水线中，作为独立 Job 与 lint 并行运行。

### 9.2 CI 流水线位置

```
push/PR
├── lint Job（现有）
├── security Job（安全审计）
│   ├── 依赖漏洞审计（pnpm audit）
│   ├── 密钥泄露检测（TruffleHog）
│   ├── 许可证合规检查（license-checker）
│   └── 安全头配置校验
├── unit-tests Job → 依赖 lint
├── build Job → 依赖 lint
└── e2e-smoke Job → 依赖 build
```

## 10. 常见问题排查

### 10.1 网络不通导致审计数据缺失

**症状**：`[1/4] 执行依赖漏洞审计...` 后提示 `⚠️ 使用缓存的审计数据`

**说明**：脚本会自动降级为读取 `docs/security/audit-raw.json` 缓存数据。这不影响其他 3 项审计的执行。如需更新缓存：

```bash
pnpm audit --json > docs/security/audit-raw.json
```

### 10.2 许可证检查工具安装超时

**症状**：`[2/4] 执行许可证合规检查...` 长时间无响应（超过 60 秒）

**解决方案**：

```bash
# 手动安装并缓存
npm install -g license-checker
# 或使用 rspack 版本
npx -y license-checker-rspack2@latest --production --summary
```

### 10.3 评分异常偏低

**症状**：安全评分低于预期，但未发现明显安全问题

**排查步骤**：

1. 检查 `summary-data.json` 中各维度扣分明细
2. 重点确认 `vulnerabilities.counts` — moderate/low 漏洞虽不阻断 CI 但仍会扣分
3. 确认 `headers.results` — 某个安全头被意外删除会扣 10 分
4. 确认 `sensitive.results` — 某个敏感模式未在 `.gitignore` 中配置

### 10.4 CI 模式下误报阻断

**症状**：`--ci` 模式下因已知无影响的漏洞导致构建失败

**解决方案**：在 `package.json` 中配置 CVE 白名单：

```jsonc
{
  "pnpm": {
    "auditConfig": {
      "ignoreCves": [
        "CVE-20XX-XXXXX", // 需注释说明忽略原因
      ],
    },
  },
}
```

### 10.5 `audit-raw.json` 格式错误

**症状**：降级读取缓存时仍无法解析

**解决方案**：

```bash
# 重新生成缓存
pnpm audit --registry https://registry.npmjs.org --json > docs/security/audit-raw.json
# 验证格式
node -e "JSON.parse(require('fs').readFileSync('docs/security/audit-raw.json','utf-8'))"
```

## 12. 维护建议

| 周期       | 活动                                                  |
| ---------- | ----------------------------------------------------- |
| 每次 PR    | 执行 `pnpm security:audit` 确认无新增高危漏洞         |
| 每周       | Dependabot 自动提交依赖更新 PR（如已启用）            |
| 每月       | 审查 `pnpm.auditConfig.ignoreCves` 白名单是否仍然合理 |
| 每季度     | 回顾 CSP 策略，评估是否可收紧 `unsafe-inline`         |
| 依赖升级时 | 运行 `pnpm security:audit` 确认许可证兼容性无变化     |
