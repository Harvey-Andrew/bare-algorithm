# 算法可视化帧生成模式设计

在本项目中，生成算法运行推演动画（Frames）时，我们遇到了针对不同复杂度数据结构的需求。为了保证动画帧的独立性（避免 JavaScript 的引用传递导致前面的帧被后续修改覆盖），我们针对具体的题目采用了两种主要的帧生成模式。

## 适用场景判定

| 特性         | 面向对象结构（如复杂链表、树、多维嵌套数组等） | 基础数据结构（如一维数组、基本类型变量指针等） |
| :----------- | :--------------------------------------------- | :--------------------------------------------- |
| **拷贝难度** | 困难（含有大量的引用、层级深）                 | 简单（解构即可 `[...nums]`）                   |
| **状态追踪** | 需要追踪节点的来源、`id`、复杂的多维度 `state` | 只需要追踪一维索引位上元素的 `state`           |
| **推荐模式** | **类状态机快照模式 (FrameGenerator)**          | **函数式不可变推帧 (createFrame)**             |

---

## 1. 类状态机快照模式：`FrameGenerator`

**典型代表**：`merge-k-sorted-lists` (合并 K 个排序链表)

### 问题痛点：

在合并 K 个链表的过程中，我们有原始的多条链表 `lists`，需要维护最小堆 `heap`，还需要将合并后的节点放入 `result` 链表。每一个节点不仅是一个值，还是一个具有特定状态（例如 `NORMAL`, `IN_HEAP`, `MERGED`）的对象引用。如果使用原生函数式写法，每次 push 帧都需要写数十行的深拷贝样板代码。

### 解决方案：

引入 `FrameGenerator` 类作为状态容器。

```typescript
class FrameGenerator {
  frames: MergeKFrame[] = [];
  lists: ListNode[][];
  result: ListNode[] = [];
  heap: ListNode[] = [];
  // ... 其他全局追踪变量

  // 核心机理：内部封装深拷贝
  getSnapshot(): MergeKFrame {
    return {
      lists: this.lists.map((l) => l.map((n) => ({ ...n }))),
      result: this.result.map((n) => ({ ...n })),
      heap: this.heap.map((n) => ({ ...n })),
      // ...
    };
  }

  // 业务逻辑只需专注算法本身
  addFrame(message: string, line: number, extra: Partial<MergeKFrame> = {}) {
    this.frames.push({
      ...this.getSnapshot(),
      message,
      line,
      ...extra,
    });
  }
}
```

**优势**：

- **隐藏复杂度**：算法主逻辑（如堆的 siftUp, siftDown）只需关注本身对 `this.heap` 和 `this.lists` 数据的操作。
- **快照自动化**：每次执行 `addFrame`，底层自动剥离和深层拷贝当前的复合状态集，使得业务代码极度清晰。

## 2. 函数式不可变推帧模式：`createFrame` 函数

**典型代表**：`kth-largest-element` (数组中的第 K 个最大元素)

### 问题痛点：

该题目只涉及一个一维数字数组 `nums`，以及配套的一维状态枚举数组 `states`，再加上诸如 `left`, `right`, `k`, `pivotIndex` 等基础类型的标量变量。在这种扁平化数据的场景下，构造一个 `Class` 进行状态管理反而显得过于冗重且过度设计。

### 解决方案：

使用一个高阶辅助函数 `createFrame`，以纯函数的方式补齐字段并推入帧。

```typescript
function createFrame(overrides: Partial<KthLargestFrame>): KthLargestFrame {
  return {
    line: 0,
    message: '',
    nums: [],
    states: [],
    k: 0,
    left: 0,
    right: 0,
    pivotIndex: -1,
    targetIndex: 0,
    result: null,
    // 基础状态通过 overrides 即时浅拷贝覆盖即可
    ...overrides,
  };
}

// 业务逻辑侧直接指明如何产生这一帧的快照
frames.push(
  createFrame({
    line: 12,
    message: `分区完成...`,
    nums: [...nums], // 扁平数组轻量解构深拷贝
    states: makeStates(...),
    left,
    right,
    pivotIndex
  })
);
```

**优势**：

- **轻量且无副作用**：符合 React 及函数式编程直觉。
- **声明式构建**：当前帧该是什么样，就传什么参数，对于不需改变的常量（如 `targetIndex`），直接赋值，数组则利用展算符 `[...arr]` 直接浅拷贝（达到数字数组的深拷贝效果）。

---

## 结论

两种模式的设计哲学都是**“确保发往渲染器的每一帧里的数据结构引用全部相互独立”**。在涉及数组指针（如二分、排序）等经典基础题目时优先应用【**函数式模式**】，以保持代码精简；而在操作链表、树、图或需要维护多级缓存结构（如复杂的动态规划表格）时，应当抽象出【**状态机快照类**】来降低开发心智负担。
