# 性能优化工作流说明

本文档描述项目当前可执行的性能优化流程，目标是把「发现问题 → 修复 → 复测 → 深挖证据」固化成标准化操作。

> **平台说明**：当前所有性能脚本均为 PowerShell (.ps1)，仅支持 **Windows** 平台。

## 1. 目标

- 快速发现首屏硬伤：大体积 JS、阻塞渲染资源、主线程阻塞
- 量化优化收益：同口径对比 `baseline` 与 `optimized`
- 留存完整证据：Lighthouse JSON、trace、HAR、截图、汇总报告

## 2. 流程总览

1. 跑 baseline 全局基准
2. 实施性能优化改动
3. 跑 optimized 全局基准
4. 生成可视化汇总报告
5. 一键自动深挖真实交互（无需人工回车）
6. 打包体积分析（Bundle Analyzer 前后对比）
7. 根据结果继续迭代

## 3. 全局基准（Lighthouse）

### 3.1 baseline

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\perf-lighthouse.ps1 -OutDir docs/performance/baseline -Port 3000
```

### 3.2 optimized

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\perf-lighthouse.ps1 -OutDir docs/performance/optimized -Port 3000
```

### 3.3 默认扫描路由

Lighthouse 脚本会自动对以下 4 条路由逐一跑分：

| 名称            | 路由                      |
| --------------- | ------------------------- |
| home            | `/`                       |
| problems        | `/problems`               |
| category-array  | `/problems/array`         |
| problem-two-sum | `/problems/array/two-sum` |

如需自定义路由，需直接修改 `scripts/perf-lighthouse.ps1` 中的 `$routes` 数组。

> [!WARNING]
> 该脚本会**强制杀掉**指定端口上正在运行的进程（包括你手动启动的 `pnpm dev`），然后自行启动和关闭 dev server。执行前请确保已保存工作。

### 3.4 基准结果校验

脚本执行后会自动校验每条路由的结果（退出码、文件是否存在、文件大小是否合理），任何失败都会立即报错终止。

跑完后请手动确认：

```powershell
# 确认生成了 4 个 JSON 文件且大小合理（通常 > 100KB）
Get-ChildItem docs/performance/baseline/*.json | Select-Object Name, Length
```

> [!IMPORTANT]
> 如果文件数量不足 4 个或文件大小异常偏小（< 1KB），说明 Lighthouse 未正常完成，请检查网络和依赖后重跑。

### 3.5 报告可视化

```shell
pnpm perf:report:visualize
```

支持自定义参数（直接调用脚本）：

```shell
node scripts/perf-report-visualize.mjs --baseline docs/performance/baseline --optimized docs/performance/optimized --outDir docs/performance/reports
```

默认输出目录：

- `docs/performance/reports/<当天日期>/summary.md`
- `docs/performance/reports/<当天日期>/summary.html`
- `docs/performance/reports/<当天日期>/summary-data.json`

> [!NOTE]
> 报告仅对比 baseline 和 optimized 的**交集路由**。如有缺失路由会输出 WARNING 提示，请确保两个目录的 JSON 文件数量一致。

## 4. 自动化真实交互深挖

### 4.1 一键命令（默认无界面）

```shell
pnpm perf:drilldown
```

该命令会自动完成：

- 启动或复用 `next dev`（默认端口 `3000`）
- 打开目标路由（默认 `/problems/array/two-sum`）
- 自动执行预设交互序列（详见下方）
- 自动导出 `trace/HAR/截图/指标`
- 自动收尾（若由脚本拉起 dev 服务，会自动关闭）

说明：不需要手工打开 DevTools，不需要在终端回车确认结束。

> [!NOTE]
> dev 模式下 drilldown 脚本会**优先复用**已在端口上运行的 dev server，不会杀掉现有进程。但 **`-Mode prod` 模式下会强制杀掉已有进程**，确保以 `next start` 提供服务。

#### 自动交互流程

脚本会模拟以下用户操作来收集性能数据：

1. `Ctrl+K` 打开搜索框 → 输入 "two sum" → 方向键选择 → Escape 关闭
2. 点击页面空白区域
3. 两次 `Space`（播放/暂停动画控制）
4. 三次 `ArrowRight`（步进动画）
5. `Home` 键回到开头
6. 鼠标滚轮下滚 720px → 上滚 420px

### 4.2 可视化观察模式（headed）

```shell
pnpm perf:drilldown:headed
```

### 4.3 自定义路由和端口（直接调用 PowerShell）

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\perf-interactive-drilldown.ps1 -OutDir docs/performance/drilldown -Port 3300 -Url /problems/graph/number-of-islands
```

### 4.4 生产模式深挖

通过 `-Mode prod` 参数可以先 `next build` 再 `next start`，减少 dev 模式编译噪声：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\perf-interactive-drilldown.ps1 -OutDir docs/performance/drilldown -Port 3000 -Mode prod
```

### 4.5 场景配置

交互场景定义在 `scripts/perf-scenarios.json` 中，通过 `--scenario` 参数加载。

使用示例（直接调用 Node 脚本）：

```shell
node scripts/perf-drilldown-auto.mjs --baseUrl http://localhost:3000 --scenario number-of-islands --outDir docs/performance/drilldown/test
```

当前预置场景：

| 场景名            | 路由                                | 说明                    |
| ----------------- | ----------------------------------- | ----------------------- |
| two-sum           | `/problems/array/two-sum`           | 数组 - 两数之和（默认） |
| number-of-islands | `/problems/graph/number-of-islands` | 图 - 岛屿数量           |

未指定 `--scenario` 时使用内置默认交互（与 two-sum 场景相同）。

### 4.6 产物目录

每次执行都会在以下路径创建带时间戳的子目录：

- `docs/performance/drilldown/<timestamp>/`

目录内产物说明：

| 文件                    | 说明                                                                |
| ----------------------- | ------------------------------------------------------------------- |
| `performance-trace.zip` | Chromium Trace 文件，可导入 DevTools Performance 面板分析           |
| `network.har`           | 完整网络请求记录，可导入浏览器 Network 面板查看                     |
| `screen-before.png`     | 交互前全页截图                                                      |
| `screen-after.png`      | 交互后全页截图                                                      |
| `auto-metrics.json`     | 自动采集的性能指标（Navigation Timing、Long Task、CLS、资源加载等） |
| `auto-summary.md`       | 本次深挖的自动摘要                                                  |
| `session-meta.json`     | 会话元信息（执行时间、端口、路由、目标 URL、执行模式）              |
| `session-notes.md`      | 备注模板，供手动填写发现和修复计划                                  |

## 5. 打包体积分析（Bundle Analysis）

通过 `@next/bundle-analyzer` 生成 Webpack 打包报告，结合自建快照 + 对比脚本，量化每次改动对产物体积的影响。

### 5.1 前置条件

- `next.config.ts` 已集成 `@next/bundle-analyzer`，通过环境变量 `ANALYZE=true` 启用
- 构建完成后会在 `.next/analyze/` 下生成 `client.html`、`nodejs.html`、`edge.html`

### 5.2 流程：Before / After 对比

#### Step 1：采集 before 快照

在修改代码**之前**，执行带分析器的构建并保存快照：

```shell
pnpm bundle:analyze:before
```

等价于：

```shell
cross-env ANALYZE=true next build --webpack   # 构建 + 生成分析报告
node scripts/bundle-snapshot.mjs before        # 保存到 docs/performance/bundle-analysis/before/<timestamp>/
```

#### Step 2：实施性能优化改动

修改代码（例如动态导入、树摇、删减依赖等）。

#### Step 3：采集 after 快照

```shell
pnpm bundle:analyze:after
```

#### Step 4：生成对比报告

```shell
pnpm bundle:compare
```

或合并 Step 3 + Step 4：

```shell
pnpm bundle:refresh:compare    # after 构建 + 快照 + 对比报告一步到位
```

### 5.3 自定义参数

```shell
# 指定自定义 before/after 目录
node scripts/bundle-compare-visualize.mjs --before docs/performance/bundle-analysis/before --after docs/performance/bundle-analysis/after

# 指定输出目录
node scripts/bundle-compare-visualize.mjs --outDir docs/performance/bundle-analysis/reports/custom
```

### 5.4 产物说明

快照保存在 `docs/performance/bundle-analysis/` 下，结构如下：

```
docs/performance/bundle-analysis/
├── before/
│   └── <timestamp>/
│       ├── client.html        # 客户端打包可视化（可浏览器打开交互查看）
│       ├── nodejs.html        # Node.js 端打包
│       └── edge.html          # Edge Runtime 打包
├── after/
│   └── <timestamp>/
│       ├── client.html
│       ├── nodejs.html
│       └── edge.html
└── reports/
    └── <timestamp>/
        ├── compare-summary.json   # 对比数据（JSON）
        ├── compare-report.md      # 对比报告（Markdown）
        ├── compare.html           # 可视化对比报告（拖入浏览器查看）
        └── bundle-compare.svg     # 摘要图表（可嵌入 PR/Issue）
```

### 5.5 对比报告内容

生成的可视化 HTML 报告包含：

- **核心指标卡片**：Initial Parsed / Gzip 总量、All Parsed / Gzip 总量、KaTeX 在 Initial 中的占比
- **入口点级别对比**：按 Entrypoint 维度展示 Parsed 和 Gzip 的 Before/After 条形图
- **Chunk 明细表**：按单个 Chunk 列出前后差异，快速定位体积变良/回归的具体模块

> [!TIP]
> 对比报告会自动选取最新的 before 和 after 快照（按时间戳排序取最新）。历史快照不会被删除，方便回溯。

## 6. 操作指南（SOP）

### 场景 A：完整一轮优化

1. `powershell -ExecutionPolicy Bypass -File .\scripts\perf-lighthouse.ps1 -OutDir docs/performance/baseline -Port 3000`
2. 校验 baseline 结果：`Get-ChildItem docs/performance/baseline/*.json | Select-Object Name, Length`
3. 完成代码优化
4. `powershell -ExecutionPolicy Bypass -File .\scripts\perf-lighthouse.ps1 -OutDir docs/performance/optimized -Port 3000`
5. 校验 optimized 结果：确认 4 个 JSON 文件且大小 > 100KB
6. `pnpm perf:report:visualize`
7. `pnpm perf:drilldown`

### 场景 B：仅更新汇总报告

```shell
pnpm perf:report:visualize
```

### 场景 C：仅跑自动深挖

```shell
pnpm perf:drilldown
```

### 场景 D：生产模式深挖（减少 dev 噪声）

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\perf-interactive-drilldown.ps1 -OutDir docs/performance/drilldown -Port 3000 -Mode prod
```

> [!WARNING]
> `-Mode prod` 会**强制杀掉**端口上的已有进程（即使是正在运行的 `pnpm dev`），然后执行 `next build && next start`。

### 场景 E：全流程一键命令

```shell
pnpm perf:full
```

该命令串联执行：baseline 扫描 → optimized 扫描 → 可视化报告 → 自动深挖。适合一次性完整评估。

### 场景 F：打包体积前后对比

```shell
# 1. 修改前拍快照
pnpm bundle:analyze:before

# 2. 实施优化...

# 3. 修改后拍快照 + 生成报告
pnpm bundle:refresh:compare

# 4. 浏览器打开 docs/performance/bundle-analysis/reports/<timestamp>/compare.html 查看
```

## 7. 验收建议

以下指标基于 **Lighthouse 移动端仿真（simulate throttling）** 的结果：

- Performance Score：核心页不低于 70
- LCP：核心页 <= 4s
- TBT：核心页 <= 300ms
- CLS：尽量接近 0
- FCP：建议 <= 3s
- SI (Speed Index)：建议 <= 5s

> [!TIP]
> 可参考 `docs/performance/baseline/` 中的 JSON 文件查看当前基准数据。

## 8. 注意事项

- `scripts/perf-lighthouse.ps1` 目前基于 `next dev`，用于同环境趋势对比。
- 若要接近线上结果，可使用 drilldown 的 `-Mode prod` 参数（自动 `next build && next start`），并建议每路由至少跑 3 次取中位数。
- 自动深挖依赖 `playwright-core`（已在 devDependencies 中），需确保本机有可用浏览器（Edge/Chrome/Chromium）。
- **Lighthouse 脚本会杀掉端口上的已有进程**；drilldown 脚本则会优先复用已有 dev server。
- Lighthouse 使用 `npx --yes lighthouse` 运行，首次执行会自动下载，无需手动安装。
- Bundle Analyzer 使用 `--webpack` flag 强制走 Webpack 打包（Next.js 16 默认使用 Turbopack）。
- 所有性能脚本依赖 PowerShell，**仅支持 Windows**；打包分析脚本为 Node.js，跨平台可用。
